## Context

`ssh-key-manager.sh` is a single flat script. This change restructures it into a subcommand-based CLI named `skey`, adds a `config` subcommand for SSH config management, and introduces a global state file to track namespaces. The script must remain bash-only with no new runtime dependencies.

## Goals / Non-Goals

**Goals:**
- Rename installed binary to `skey`
- Dispatch to `skey create` / `skey config` / `skey help` via a subcommand router
- `skey config` generates a YAML file at a target folder and upserts a managed block in `~/.ssh/config`
- `~/.ssh/.skey.yaml` tracks registered namespaces and their config file paths
- Keep everything bash-only — no `yq`, no Python, no additional runtime deps

**Non-Goals:**
- Key deletion, rotation, or listing
- Parsing or modifying existing manual entries in `~/.ssh/config`
- Multi-host entries per namespace (one `Include` per namespace)
- Validating or testing the generated SSH config against live hosts

## Decisions

**Decision: Single script file, subcommand dispatch via `case`**
Rationale: Keeps the tool as one auditable file. A `case "$1"` router at the top dispatches to functions `cmd_create`, `cmd_config`, `cmd_help`. Alternative (separate files per subcommand) rejected as over-engineering for a personal tool.

**Decision: Inline YAML generation with `printf` / heredoc — no `yq`**
Rationale: `yq` is not universally installed. The YAML structure for SSH config is shallow and predictable; `printf` is sufficient. Reading `.skey.yaml` for state can use `grep` patterns since the schema is fixed.

**Decision: `~/.ssh/.skey.yaml` as global state**
Rationale: Central place to know which namespaces exist and where their config YAMLs live. Dot-prefixed to keep it out of the way in `~/.ssh/` listings. Schema:
```yaml
namespaces:
  work:
    config: ~/.ssh/work/.skey-config.yaml
  personal:
    config: ~/.ssh/personal/.skey-config.yaml
```

**Decision: Managed block in `~/.ssh/config` using sentinel comments**
Rationale: Allows safe idempotent updates. The block is bounded by:
```
### generated by skey ###
...
### end generated by skey ###
```
On each `skey config` run, the block is replaced in-place (or appended if absent). Everything outside the block is untouched.

**Decision: Per-namespace config YAML lives inside the namespace directory**
Rationale: Keeps config co-located with the keys it describes. Path: `~/.ssh/<namespace>/.skey-config.yaml`. This is the "target folder" the user specifies (defaults to `~/.ssh/<namespace>/`).

**Decision: `skey config` registers the namespace in `.skey.yaml` and regenerates the `~/.ssh/config` block**
Rationale: Single command that both creates the per-namespace config and applies it globally. Running it again is idempotent.

## Risks / Trade-offs

- `~/.ssh/config` manipulation → corrupting SSH config breaks all SSH. Mitigation: write to a temp file, validate it opens without error (`ssh -G` check), then replace. If validation fails, keep original.
- Concurrent writes to `~/.ssh/config` → unlikely for a personal tool; no locking needed.
- Grepping `.skey.yaml` with bash → fragile if the file is hand-edited in unexpected ways. Mitigation: document the schema, fail gracefully with a clear error if parsing fails.

## Migration Plan

- Existing `ssh-key-manager` installs: re-run `install.sh` to get `skey`. Old binary is not removed automatically — user deletes `~/.local/bin/ssh-key-manager` manually.
- No changes to existing key files under `~/.ssh/`.
